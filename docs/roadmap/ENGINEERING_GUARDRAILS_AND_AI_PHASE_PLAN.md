# PACKLOG 工程护栏与 AI 接入分阶段方案

## 目标

- 数据模型少而稳：主实体少，事件记录完整，可多维聚合。
- 前端组件统一：同一类交互在各页面一致，不写死。
- 多语言统一：所有文案走统一 i18n 决策链，避免分叉逻辑。
- 统计可追溯：所有统计来自用户真实操作日志，不硬编码。

## 一、数据层护栏（必须长期坚持）

1. **主实体唯一**
   - `gear_master` 作为全局装备实体（跨 trip / locker / community）
   - 其他表只做“场景关系”或“实例快照”

2. **事件优先，快照其次**
   - 评价、使用、重量变化属于事件数据，写入日志表，不覆盖历史
   - 页面展示可读取“最新快照 + 历史事件聚合”

3. **字段命名和来源一致**
   - 重量统一 `weight_g`
   - 来源统一 `weight_source`（`user | library | estimated`）
   - 复盘统一 `review_result / review_utility / review_note`

4. **兼容迁移策略**
   - 新字段上线必须可空
   - 应用层先做列存在探测，迁移后再切默认逻辑

## 二、前端层护栏（组件化与一致性）

1. **同一动作同一组件**
   - 复盘控件（结论 + 星级 + 备注）使用统一交互组件
   - 重量输入与提示在新增/编辑入口统一

2. **即时反馈优先**
   - 先本地状态更新，再后台写入（失败回滚）
   - 避免整页刷新，优先局部刷新或软刷新

3. **统计只读聚合**
   - 前端不写死“xx 件/xx%”
   - 使用实时聚合结果渲染

4. **多语言不可分叉**
   - UI 文案全部走 `pickLangText`
   - 动态文本走 `resolveLocalizedText`

## 三、AI 接入建议（下一阶段）

### Phase AI-1（低风险，高收益）

- **重量建议助手**
  - 输入名称时给 `weight_g` 建议与置信提示
  - 结果不直接覆盖，用户可确认修改

- **复盘摘要助手**
  - 从 `gear_trip_usage_logs` 生成“这次最值得保留/淘汰项”
  - 明确标注“基于你的历史数据生成”

### Phase AI-2（中风险）

- **场景差异建议**
  - 根据目的地/季节/历史 trip 对比，推荐增减项
  - 使用规则 + 模型混合，保留可解释依据

### Phase AI-3（高价值，需治理）

- **社区模板质量评分**
  - 结合复制率、加购率、复盘有效率形成质量分
  - 模型只做辅助评分，最终展示可解释指标

## 四、当前实现状态（本次迭代）

- 已将行程页接入重量分布统计展示（基于用户条目重量）

## 五、后续执行建议

2. 再做复盘历史在装备库中的统一展示
3. 最后接 AI 层，严格走“建议不替用户决策”的产品原则
